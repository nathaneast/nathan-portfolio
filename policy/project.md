## 프로젝트 개발 순서
- `spec/prd.md` 파일을 분석하고 유저스토리에 따라 작업 task들을 생성하고 확인을 받는다.
  - 기능 개발은 페이지, 화면 단위로 개발을 진행한다. (레이아웃, 유저 인터랙션, 디자인 포함)
  - 병렬 에이전트로 업무를 지시하기 효율적이도록 업무 범위, 성격이 중복되지 않도록 고려한다.
- task 확정시 `task/todo.md` 파일을 생성하고 1개씩 작업을 진행하고 완료시 체크한다.
- task 작업이 완료되거나, 개발 서버에서 확인이 필요한경우 `dev` 인프라 셋업 및 배포한다.
- 버그 수정, 추가 개발
- 테스트
- 기능 개발이 모두 완료되면 `prod` 동일하게 셋업하고 배포 


## 프로젝트 진행
- 기획, 유저스토리가 수정, 변경되는경우 `spec/prd.md` 파일에서 특정 부분을 찾아서 수정하여 업데이트 한다.


## 자동화
- git, convex, vercel, claudeflare 등 과 같은 대시보드에서 프로젝트를 설정하고 값을 입력, 셋팅해야하는 작업은 mcp를 적극적으로 활용해 작업하고 사용자의 수동 작업을 최소화 한다.


## 브랜치 전략
- `dev`: 개발서버 브랜치, `main`: 운영서버 브랜치
- 항상 `dev` 브랜치를 사용하고 commit, push 모두 `dev` 브랜치만 진행한다.
- `mian` 브랜치는 dev브랜치와 병합만 한다. `main` 브랜치는 항상 dev 브랜치와 병합후에 dev 브랜치로 체크아웃 한다.


## 배포 환경 운영
- dev: 개발서버 브랜치, main: 운영서버 브랜치
- 환경은 `dev`(개발)와 `prod`(운영)로 분리해 운영한다.
- 개발은 반드시 `dev` 환경에서 먼저 진행하고 검증한다.
- 운영 배포는 `dev` 검증 완료 후 `prod`로 승격한다.
- 브랜치 전략은 `dev`에서 개발을 진행하고, 작업 확정 후 `main` 브랜치로 병합한다.
- 브랜치 기준은 `dev` -> `main` 승격 흐름을 따른다.
- 모든 작업 커밋/푸시는 반드시 `dev` 브랜치에서만 수행한다.
- `main` 브랜치로의 직접 커밋/직접 푸시는 금지한다.
- `main` 반영은 `dev`에서 품질 게이트(`lint`, `typecheck`, 핵심 테스트) 통과 후 승격 병합으로만 허용한다.
- Vercel 프로젝트는 `dev`/`prod`를 분리하고, 깃허브 푸시가 있을시 자동 배포된다.
- 도메인/환경변수는 `dev`/`prod`를 전부 분리한다.
- `local` / `dev` 환경에서 실행될때에는 현재 환경을 인식 가능하도록 왼쪽 상단에 뱃지 ui로 표시한다.
